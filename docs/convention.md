# 우리 팀 Task 관리 전략

> Issue · Branch · PR · Commit 운영 규칙. 모든 협업은 이 문서를 기준으로 한다.

---

## 1️⃣ Issue 관리 전략

Issue는 **명확한 완료 기준(Definition of Done)을 정의할 수 있는 작업**을 기준으로 생성한다.

- 작업 진행 상황을 쉽게 추적할 수 있도록 최대한 **세분화**하여 관리한다.
- 하나의 Issue는 **하나의 목표**를 가지도록 작성한다.

### ❓ 하위 Issue가 너무 많아지는 경우

- 관리 비용이 과도하게 증가할 경우, **관련성이 높은 작업들을 묶어서** 관리한다.
- 주제 또는 기능 단위로 **2~4개의 Issue를 하나의 상위 작업**으로 구성할 수 있다.

### 계층 구조

```
상위 Issue (Epic / 모듈·기능)      ← 추적용 컨테이너, Branch 없음
  └ 그룹 Issue (관련 작업 묶음)     ← 2~4개 묶을 때 (선택)
      └ 작업 Issue                ← 실제 작업 단위 = 1 PR ★
          └ 체크리스트 (- [ ])     ← Issue 본문, commit 단위
```

---

## 2️⃣ Branch 생성 기준

Issue의 규모가 충분히 크다면 **Issue별로 Branch를 생성하여 작업**한다.

- 작업 완료 후 Pull Request를 통해 **`dev` 브랜치에 병합**한다.
- 병합이 완료된 Branch는 **삭제**한다.

Issue의 규모가 작다면, 관련된 **2~4개의 Issue를 하나의 Branch로 묶어서** 작업할 수 있다.

### 브랜치 네이밍

```
<type>/<이슈번호>-<짧은-설명>

feat/6-signup
fix/27-token-expiry
docs/5-readme
```

- 분기 대상: **`dev`** (통합 브랜치)
- `main`은 안정/배포 라인 — 직접 작업하지 않는다.

---

## 3️⃣ PR (Pull Request) 기준

우리 팀은 **1 Branch = 1 PR** 원칙을 사용한다.

- 하나의 Branch는 **하나의 작업 목적**만 가지도록 유지한다.
- PR에는 해당 Branch에서 수행한 작업만 포함한다.
- 리뷰 및 이력 추적이 용이하도록 **불필요하게 많은 작업을 하나의 PR에 포함하지 않는다.**
- PR 본문에 `Closes #이슈번호` 를 적어 머지 시 Issue가 자동으로 닫히게 한다. (여러 개면 `Closes #6, Closes #7`)
- 병합 대상은 **`dev`**.
- 병합 방식은 **Merge commit 또는 Rebase**를 사용한다. **Squash 금지** — 커밋 단위 이력(이슈 ↔ 커밋 연결)을 보존하기 위함.

### 한 PR 크기 테스트 — 묶기/나누기 기준

> **이 묶음이 한 PR(~400줄, 1~3일, 응집된 변경)에 들어가는가?**
>
> - **YES** → 관련 Issue 2~4개를 1 Branch = 1 PR로 묶는다.
> - **NO** → 한 그룹이라도 2개 이상의 PR로 나눈다.

- 그룹(상위로 묶인 Issue)은 **추적 단위**일 뿐, 자동으로 PR 경계가 아니다.
- 작고 응집적인 묶음(예: 회원가입+로그인)은 1 PR로 묶고, 각 항목이 독립적으로 큰 묶음(예: 게시글/댓글/태그/검색)은 나눈다.
- 큰 PR은 리뷰가 형식적(고무도장)이 되므로 **작게 유지**한다.

## 4️⃣ Commit 컨벤션

### 타입

| 타입       | 내용                                                  |
| ---------- | ----------------------------------------------------- |
| `feat`     | 새로운 기능에 대한 커밋                               |
| `fix`      | 버그 수정에 대한 커밋                                 |
| `build`    | 빌드 관련 파일 수정 / 모듈 설치 또는 삭제에 대한 커밋 |
| `chore`    | 그 외 자잘한 수정에 대한 커밋                         |
| `ci`       | CI 관련 설정 수정에 대한 커밋                         |
| `docs`     | 문서 수정에 대한 커밋                                 |
| `style`    | 코드 스타일 혹은 포맷 등에 관한 커밋                  |
| `refactor` | 코드 리팩토링에 대한 커밋                             |
| `test`     | 테스트 코드 수정에 대한 커밋                          |
| `perf`     | 성능 개선에 대한 커밋                                 |

### 커밋 메시지 형식

```
<type>: <변경 내용 요약> (#이슈번호)

예) feat: JWT access 토큰 발급 로직 추가 (#7)
```

### 커밋 기준

- **이슈의 할 일 목록(체크리스트) 하나마다 commit 생성.**
- 체크리스트 작업 하나가 끝날 때마다 해당 내용에 대한 commit을 작성한다.
- 커밋 메시지 끝에 **이슈 번호 `(#N)`** 를 붙여 커밋 ↔ 이슈를 자동 연결한다. (GitHub 이슈 타임라인에 커밋이 표시되어, PR에 매핑 표를 따로 쓰지 않아도 추적 가능)
