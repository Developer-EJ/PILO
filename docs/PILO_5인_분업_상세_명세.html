PILO 5인 분업 상세 명세

인프라는 별도 담당으로 분리하고, 5명의 개발자가 제품 도메인 단위로 기능을 나누어 개발하기 위한 역할 명세서입니다.

공통 원칙

  • 각 담당자는 자기 도메인의 화면, API, DB 모델, 비즈니스 로직, 테스트를 책임진다.
  • 서버 단위가 아니라 도메인 단위로 분리한다.
  • 다른 도메인의 데이터를 직접 수정하지 않고, 해당 도메인의 API 또는 계약을 통해 연동한다.
  • 세인은 공통 Agent Runtime과 Orchestrator를 담당한다.
  • 동현, 주형, 진호, 은재는 자기 기능에서 필요한 Agent Workflow를 담당한다.

동현: Auth / Workspace / Dashboard / Canvas

담당 목표

동현은 사용자가 PILO에 로그인하고, 가장 먼저 마주하는 프로젝트 작업 공간을 담당한다.

핵심 역할은 사용자를 식별한 뒤 프로젝트를 한눈에 보고, 주요 기능으로 진입하고, 전체 구조를 시각적으로 이해할 수 있게 만드는 것이다.

주요 기능

  • 로그아웃

GitHub 로그인은 사용자 인증 수단이다. GitHub Repository 연결, Issue/PR 조회, Webhook, GitHub App 설치는 주형의 GitHub 연동 도메인으로 분리한다.

Dashboard

Canvas

Canvas는 직접 업무를 처리하는 곳이라기보다 프로젝트 구조를 시각화하는 영역이다.

단, Canvas는 단순한 관계도 화면이 아니라 Miro/FigJam처럼 넓은 보드 위에서 프로젝트 객체를 배치하고 정리하는 개발 협업 공간으로 구현한다. PILO의 Canvas는 자유 그림판이 아니라 Task, 회의록, PR, GitHub Issue, 문서, 파일, 코드, 의사결정, 리스크 같은 프로젝트 운영 객체를 카드 형태로 올려두고 관계를 시각화하는 구조화된 캔버스다.

  • 무한 캔버스 기반 화면
  • Canvas 줌 및 이동
  • 프로젝트 객체 카드 배치
  • Task 노드 표시
  • 회의록 노드 표시
  • PR 노드 표시
  • GitHub Issue 노드 표시
  • 문서 및 파일 노드 표시
  • 코드 파일 노드 표시
  • 의사결정 노드 표시
  • 리스크 노드 표시
  • 카드 드래그 이동
  • 카드 크기 조절
  • 노드 간 관계 표시: 회의록 -> 생성된 Task, Task -> GitHub Issue, Task -> PR, PR -> Code Review Room
  • 파일/코드 -> 관련 Task, 리스크 -> 관련 Task 또는 PR, 의사결정 -> 관련 회의록 또는 Task 관계 표시
  • 노드 클릭 시 상세 패널 표시
  • 필터 제공: Task만 보기, Meeting만 보기, PR만 보기, 특정 담당자 기준 보기, 지연 항목만 보기, 리스크 항목만 보기
  • 노드 위치 저장
  • Canvas view 설정 저장: zoom, viewport position, filter 상태
  • 자동 배치 기본 지원

Canvas는 실제 업무 데이터를 직접 소유하지 않는다. Canvas가 소유하는 것은 화면 배치와 보기 설정이다.

  • CanvasBoard
  • CanvasShape
  • CanvasConnection
  • CanvasNodePosition
  • CanvasViewSetting
  • CanvasFilterSetting

Task, Meeting, PR, GitHub Issue, Report, Code Review 데이터 자체는 각 담당 도메인이 소유한다. Canvas는 해당 데이터를 read model 또는 public contract로 받아 카드 형태로 표시한다.

Canvas의 시각 객체와 실제 프로젝트 데이터는 분리한다.

Canvas Shape

  • 캔버스 위에 보이는 카드 또는 연결선
  • 위치, 크기, 색상, 접힘 여부, 표시용 제목, entityId를 가진다

Project Entity

  • 실제 업무 데이터
  • Task, Meeting Report, PR, GitHub Issue, Decision, Risk, File 등
  • 각 도메인의 DB와 API가 소유한다

Canvas Shape는 entityType, entityId를 통해 실제 Project Entity를 참조한다.

다른 도메인은 Canvas Shape를 직접 생성하거나 수정하지 않는다. 다른 도메인은 Canvas에 표시할 요약 데이터와 연결 ID를 public contract 또는 read model로 제공하고, 동현이 이를 Canvas Shape와 연결선으로 변환한다.

MVP Canvas에서는 다음 기능을 후순위로 둔다.

  • Task 생성 및 상태 변경 로직
  • Meeting Report 생성 로직
  • PR diff 분석 로직
  • GitHub API 동기화 로직
  • AI Agent 실행 로직
  • 실시간 다중 편집
  • 자유 펜 드로잉 고급 기능
  • 댓글/투표/스티커 전체 화이트보드 기능

소유 데이터

  • User
  • OAuthAccount
  • AuthSession
  • Workspace
  • WorkspaceMember
  • DashboardPreference
  • CanvasBoard
  • CanvasShape
  • CanvasConnection
  • CanvasNodePosition
  • CanvasViewSetting
  • CanvasFilterSetting

Task, Meeting, PR, GitHub Issue, Report, Code Review 데이터 자체는 각 담당 도메인이 소유한다. 동현은 해당 데이터를 public contract 또는 read model로 받아 조회해서 보여주는 역할을 맡는다.

연결 및 경계

주형에게 Task, GitHub Issue, Progress 데이터를 받고, 진호에게 Meeting Report를 받으며, 은재에게 PR Analysis 요약을 받는다. 세인에게서는 Agent 추천 액션과 프로젝트 계획 요약을 받는다.

Task 생성 및 상태 변경 로직, GitHub API 연동, 회의록 생성 로직, PR 코드 분석 로직, Agent 실행 로직은 직접 구현하지 않는다.

주형: Task / GitHub / Progress

담당 목표

주형은 PILO의 실제 작업 관리와 GitHub 연결을 담당한다. 핵심 역할은 프로젝트가 실제로 얼마나 진행되고 있는지 계산하고, Task와 GitHub Issue 및 PR을 연결하는 것이다.

주요 기능

  • Task 생성, 수정, 삭제, 상세 조회, 목록 조회
  • 담당자, 상태, 우선순위, 마감일 관리
  • Task 설명, 체크리스트, 댓글, 활동 로그
  • Task와 마일스톤 연결, Task 간 의존성 설정
  • Agent가 생성한 Task 후보 승인 및 거절
  • GitHub Repository 접근 권한 연결
  • GitHub App 설치 또는 Repository scope 승인
  • Repository 선택 및 정보 저장
  • Task에서 GitHub Issue 생성
  • GitHub Issue와 Task 동기화
  • PR 목록 조회 및 PR과 Task 연결
  • GitHub Webhook 수신
  • 전체, 마일스톤별, 담당자별 진행률 계산
  • 지연 Task, Blocked Task, 리뷰 대기 Task 탐지

소유 데이터

  • Task
  • TaskStatus
  • TaskComment
  • TaskDependency
  • Milestone
  • GitHubConnection

Agent Workflow

  • 회의록에서 나온 Action Item을 Task 후보로 변환

  • GitHub Issue 제목 및 본문 자동 생성

연결 및 경계

동현에게 Dashboard 및 Canvas용 Task 요약을 제공하고, 진호에게 회의록 기반 Task 생성 API를 제공한다. 은재에게는 PR과 연결된 Task 정보를 제공하며, 세인에게 Task 생성 및 수정 Action Contract를 제공한다.

Canvas 시각화, 로그인/회원가입 및 사용자 세션 관리, 회의 음성 및 회의록 생성, PR 코드 분석, 공통 Agent Runtime은 직접 구현하지 않는다.

세인: Multi-Agent Orchestrator / Agent Runtime / Project Planning

담당 목표

세인은 PILO의 AI Agent 시스템 전체의 공통 기반을 담당한다. Agent 호출, Context 구성, Action 반환, 사용자 승인, 실행 로그, 도메인별 Workflow 표준화를 설계한다.

연결 및 경계

동현에게 Agent 추천 액션을 제공하고, 주형에게 Task 생성 및 진행률 분석 Action을 전달한다. 진호에게 회의록 요약 Workflow 실행 구조를 제공하고, 은재에게 PR 분석 Workflow 실행 구조를 제공한다.

Workspace 화면, Task CRUD 전체, GitHub 연결 전체, 회의 UI, PR Diff Viewer, Canvas UI는 직접 구현하지 않는다.

기능 간 경계

Project Start Flow

  • 세인: 질문, 추천, 계획 생성
  • 동현: 온보딩 화면과 결과 표시
  • 주형: 생성된 Task 및 Milestone 저장
  • 진호: 첫 회의 아젠다 연결
  • 은재: 관여 없음

Meeting -> Task Flow

  • 진호: 회의 진행, 회의록 생성, Action Item 추출
  • 세인: Agent Workflow 실행과 Action Contract 변환
  • 주형: Task 후보 저장, 승인, 생성
  • 동현: Dashboard 및 Canvas에 표시
  • 은재: 관여 없음

Task -> GitHub Issue Flow

  • 주형: Task와 Issue 생성 및 동기화
  • 동현: Dashboard 및 Canvas 표시
  • 세인: Issue 본문 초안 생성 Agent 지원 가능
  • 진호, 은재: 관여 없음

PR Review Flow

  • 주형: PR 목록과 Task 연결 정보 제공
  • 은재: PR 분석과 Code Review Room
  • 세인: PR Analysis Agent 실행 구조 제공
  • 동현: 리뷰 필요 PR 요약 표시
  • 진호: 관여 없음

Project Progress Check Flow

  • 주형: 진행률 계산
  • 동현: Dashboard 표시
  • 세인: 다음 액션 추천
  • 진호: 최근 회의록 반영
  • 은재: 리뷰 병목 반영

결론

이 분업은 기능이 서로 겹치지 않으면서도 PILO의 핵심 흐름을 모두 커버한다.

동현은 사용자가 프로젝트를 보고 진입하는 작업 공간을 만들고, 주형은 작업과 GitHub를 실제로 움직이는 기능을 만든다. 진호는 회의에서 정보를 뽑아 업무 데이터로 연결하고, 은재는 PR을 이해하고 리뷰할 수 있는 구조를 만든다. 세인은 Agent가 안정적으로 일하게 만드는 실행 기반을 만든다.

이 기준을 지키면 각자가 동시에 개발해도 서로 침범할 가능성이 낮다.
